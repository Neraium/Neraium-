Technical Overview

Current Technical Model and Evidence Boundary

This page describes Neraium’s current technical framing in a way that stays inside what can be represented cleanly. The platform is being built as a read-only infrastructure intelligence layer for telemetry ingestion, deviation analysis, and reviewable records. Current public evidence is best framed as benchmark-based analytical work, not as completed field validation in water infrastructure or any other live operating environment.

01
Boundary

System Boundary

The current technical boundary is intentionally narrow. Neraium is framed as a read-only layer that receives telemetry, runs analytical logic, preserves reviewable records, and supports human interpretation. It does not issue control commands, adjust equipment states, or claim autonomous intervention.

BND-001
Ingest Only
Source telemetry is treated as inbound data. The technical model assumes data intake without outbound operating commands.
BND-002
Analyze Only
Analytical logic is limited to reviewing signal behavior, identifying persistent deviation, and generating reviewable output.
BND-003
Review Only
Outputs are meant to support human review and operator judgment, not replace operational decision-making.
BND-004
No Actuation
No setpoint writes, no direct equipment control, no autonomous remediation, and no claim of physical intervention.
BND-005
Authority Retained
Operators, licensed personnel, and authorized stakeholders retain full authority over interpretation and response.
BND-006
Record Preservation
The intended system design includes preserving source context, analysis context, and review context in structured form.

03
Benchmark

Current Benchmark Basis

The current public analytical basis can reference NASA C-MAPSS. That reference is useful for algorithm development and sequence analysis, but it must be framed correctly.

Source Dataset
NASA C-MAPSS, a public turbofan engine degradation benchmark commonly used for run-to-failure prognostics and time-series analysis.
Useful For
Testing multivariate degradation logic, baseline comparison methods, sequence scoring, and lead-time oriented signal review under controlled benchmark conditions.
What It Represents
A benchmark environment for algorithm demonstration. It supports technical method development, not direct proof of live infrastructure performance.
What It Is Not
It is not water infrastructure telemetry. It does not, by itself, establish validated performance in water systems, facilities, or operator workflows.
Correct Framing
Current results are best described as benchmark-based analytical work, prototype logic validation, and evidence of technical direction.
04
Workflow

Reference Analysis Workflow

A clean technical description of the current reference workflow can stay at the level below. It explains how analysis is structured without claiming finalized production implementation details beyond what is publicly defensible.

REF-001
Sequence Intake
Accept time-series sequences, segment them into analysis windows, and retain the relationship between raw inputs and downstream derived outputs.
REF-002
Baseline Comparison
Compare current signal behavior to prior or baseline behavior so the system can identify persistent divergence rather than isolated single-point anomalies.
REF-003
Multivariate Scoring
Use multiple correlated signals together so emerging deviation can be evaluated as a sequence pattern, not only as a threshold breach in one channel.
REF-004
Review Packaging
Produce a reviewable output that keeps enough context for an operator or technical reviewer to understand what changed and why it was flagged.
REF-005
Record Context
Preserve associated metadata such as timing, signal scope, and analysis state so that later review does not depend on memory or unsupported reconstruction.
REF-006
Human Decision Path
Keep final interpretation and action external to the analytical layer so the model remains read-only and the operational authority boundary stays intact.
05
Validation

Validation Path

The next factual step is not broader language. It is domain-specific validation on relevant infrastructure telemetry where the observed result can be tied directly to the operating environment being served.

VAL-001
Benchmark Stage
Use public benchmark data to test analytical logic, sequence handling, and deviation scoring behavior in a controlled context.
VAL-002
Domain Data Stage
Apply the same analytical framework to relevant infrastructure telemetry where the signals and operating conditions match the intended deployment domain.
VAL-003
Bounded Review Stage
Evaluate outputs with real operators or technical reviewers so the result can be assessed against actual operating context and decision pathways.
